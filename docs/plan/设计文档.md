## 文档概要

### 产品设计文档 (`docs/design/product.md`)

- **产品名称**: RustLLM Gateway
- **核心概念**: Input/Virtual/Output Node 构成的有向图路由网表
- **协议策略**: Tier 1 (OpenAI/Anthropic)、Tier 2、Tier 3 分层支持
- **差异化功能**: 负载热迁移（Phase 2 非流式、Phase 3 流式）
- **配置方式**: TOML + CLI，支持热加载

### 开源竞品调研 (`docs/research/competitors.md`)

- **主要竞品**: LiteLLM (Python), Bifrost (Go), Helicone (Rust), Kong AI
- **功能对比**: 协议支持、负载均衡、性能指标
- **市场机会**: Rust 性能优势、图结构路由差异化、分层协议策略

### 技术架构设计 (`docs/design/architecture.md`)

- **设计哲学**:
  - Input Node = HTTP Server (hyper)
  - 协议动态识别 (path → body 字段)
  - RoutingPayload = `path + headers + content: Value + visited_nodes: Vec<Arc<Node>>`
  - 健康感知内置，DFS+ 回溯
  - 图结构纯粹，无 Edge 类型 (`targets: Vec<Arc<Node>>`)
  - 策略封装权重

- **核心数据结构**:
  - `Node`: 统一枚举 (Input/Virtual/Output)
  - `VirtualNode`: `targets: Vec<Arc<Node>>`，权重在策略内部
  - `HealthRegistry`: 全局健康状态注册表

- **路由策略**:
  - `RoundRobinStrategy`: 轮询
  - `WeightedStrategy`: 权重（内部维护）
  - 健康检查内置，DFS 自动跳过不可用节点

### 开发路线图 (`docs/plan/roadmap.md`)

- **开发方法**: TDD 测试驱动开发 + 文档先行

- **Phase 1 (MVP)**: 1.5 周
  - 协议调研 → `docs/protocols/openai.md`, `anthropic.md`
  - 设计接口 → 编写测试 → 实现功能
  - 核心代理、Tier 1 协议、健康感知 DFS、基础统计

- **Phase 2 (功能完善)**: 1.5 周
  - 高级负载均衡、非流式热迁移、Tier 2 协议、CLI 增强

- **Phase 3 (生产就绪)**: 1.5 周
  - 限流、HTTP API、JWT 认证、分布式追踪

- **Phase 4 (生态建设)**: 2 周
  - 插件系统、语义缓存、多区域集群、流式热迁移

**总计**: 6.5 周完成全部开发

## TDD 开发流程

```text
对于每个功能模块：

1. 文档准备（协议调研/技术调研）
       ↓
2. 设计接口（定义 Trait/Struct）
       ↓
3. 编写测试（此时编译失败或测试失败）
       ↓
4. 实现功能（测试通过）
       ↓
5. 重构优化
```
